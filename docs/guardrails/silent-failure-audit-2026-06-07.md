## Guardrail Rule

**Observable Rejection Rule:** If a player clicked, selected, used, cast, bought, studied, equipped, or targeted something, the rejection must be observable (emit `PLAYER_ACTION_REJECTED` or appropriate event).

**Intentional No-Op Categories** (no observable event needed):
1. **Type-guard dispatch failures** — Expected internal failures in command routing (document as architectural pattern)
2. **Condition not met** — Internal pre-conditions evaluated but not player actions (e.g., passive ability checks)
3. **Race condition prevention** — Legitimate safety guards for concurrent game state changes
4. **Autonomous system guards** — Non-player actions (enemy turn, status application, respawn logic)
